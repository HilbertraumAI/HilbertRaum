## 1. Executive summary

### Overall health

The app is in **very good shape** — measurably better than any prior round found it. This is
the first full audit of this repo in which **no Critical or High finding survived
verification**. The four July audit rounds plus the post-launch tester waves have left the
core — vault/crypto, offline enforcement, the RAG pipeline, IPC contracts, the i18n system,
the test suite, CI/packaging — demonstrably solid:

- **Security:** an entire dedicated pass over the vault, network guard, download chain, skills
  sandbox, manifest→spawn trust chain, SQL/FTS injection surfaces, and IPC validation produced
  exactly **one Low finding** (F-39, CSP `style-src 'unsafe-inline'` — a defense-in-depth nit).
  The hard rules (no cloud, no telemetry, offline-usable, user data local) were re-verified
  clean, including empirical re-testing of the numeric-IP SSRF canonicalization.
- **Contracts:** all 111 IPC channels are symmetric between preload and main with zero orphans;
  the EN/DE catalogs are in exact 1,398-key parity with zero interpolation mismatches.
- **Tests:** the suite's own hygiene claims held up under spot-checking (the TS-1 "no unjustified
  fixed sleeps" claim passed 6/6 spot checks; zero `.only`/unconditional `.skip`; assertion
  vacuousness ~5/4,056 it-blocks, all env-gated harness printers).

The residual defect mass has moved **outward from the core** to four clusters:

1. **Sidecar parity gaps** — hardening applied to one sidecar but never ported to its siblings
   (F-14: the translation runtime's crash-recovery fix missing in vision; F-32: the engine
   install guard covering only the chat runtime).
2. **Shell-script parity and DIY-drive assembly** — the PowerShell/bash sibling scripts have
   drifted at exactly the spots the repo's own portability lessons (M23 etc.) didn't reach
   (F-03, F-04, F-18, F-19), and **every DIY `--with-assets` drive ships without OCR data**
   (F-05, the root cause behind open issue #59).
3. **Model-catalog data hygiene after the fast-moving Qwen promotion** — the promotion's tier
   tables landed numerically exact, but a manifest with an incoherent ctx/RAM pair (F-06),
   mixed GiB/decimal-GB units (F-16), stale license-review notes (F-17), and — most importantly —
   **the promotion decision-record misquoting the incumbents' eval numbers** (F-08) slipped
   through the promotion's own test pins.
4. **Docs that lag promotions** — docs/benchmark.md still documents the pre-promotion
   recommendation mapping as live (F-07), model-policy/model-benchmarks still carry
   present-tense "rank 0 / never auto-recommended" claims for promoted models (F-09), and
   data-contracts.md regressed a prior audit fix on the manifest count (F-20).

#### F-01 — Structured-extract scan cache is content-keyed only — a chat-model swap never refreshes extractions, even on an explicit re-run

**Medium · confidence High · likely-bug · dimension: backend-pipeline · verifier: CONFIRMED**

- **Locations:** `apps/desktop/src/main/services/analysis/extract.ts:103-111` · `apps/desktop/src/main/services/analysis/extract.ts:217-224` · `apps/desktop/src/main/services/analysis/extract.ts:297-326` · `apps/desktop/src/main/services/analysis/extract.ts:79-96` · `apps/desktop/src/main/services/analysis/extract.ts:328-332` · `apps/desktop/src/main/services/analysis/tree-build.ts:29-39` · `apps/desktop/src/main/services/analysis/tree-build.ts:216-224` · `apps/desktop/src/main/services/doctasks/handlers/tree.ts:57-83` · `apps/desktop/src/renderer/screens/documents/DocRow.tsx:142-143` · `apps/desktop/src/renderer/screens/DocumentsScreen.tsx:561-574`
- **Relation to known/tracked items:** null — no match in docs/known-limitations.md (grep for model-swap/content-keyed/extraction is clean), no row in architecture.md §46-§49 ledgers, no BUILD_STATE §5/§6 open item. Nearest text is BUILD_STATE.md:88-89 (#50 entry), which names the tree cache's model-keying as the posture extract 'now matches' but dispositions only the empty-generation half — it does not track this gap.

**Description.** The per-chunk extract cache key is sha256(chunk text + NUL + TYPE_SET_VERSION) with no model id, and the cache-hit lookup (markerExists) matches (chunk_id, '__scan__', hash, 'ok') with no model_id predicate. After the user switches to a different (e.g. much stronger) chat model, every previously scanned chunk still hits the old-model 'ok' marker, so an explicit re-run of the extract task makes ZERO model calls and flips extract_status to 'ready' without re-extracting anything. There is no path to refresh extractions to the new model short of re-indexing the document (which mints new chunk ids). A pass resumed across a model swap also produces a silently mixed-model extraction_records set (rows record model_id, but aggregation ignores it). This is exactly the invalidation dimension the sibling tree build handles: its summary_cache is keyed by (content_hash, model_id) with the M12 rationale that 'a model-switch can't yield a mixed-model tree'. The extract module documents text-change and type-set-change invalidation but is silent on model change, suggesting an oversight rather than a decision.

**Evidence.** extract.ts:110 `return createHash('sha256').update(`${text}\u0000${TYPE_SET_VERSION}`).digest('hex')` — no model id. extract.ts:221-224 `SELECT 1 FROM extraction_records WHERE chunk_id = ? AND record_type = ? AND content_hash = ? AND normalized_value = ? LIMIT 1` — no model_id column in the predicate, while deps.modelId is described only as 'recorded on each row; cache is content-keyed'. Contrast tree-build.ts:216-218 `SELECT summary_text FROM summary_cache WHERE content_hash = ? AND model_id = ?` and tree-build.ts:38-39 'a model-switch can't yield a mixed-model tree (the cache is model-keyed — M12)'.

**Verifier assessment.** Every mechanical claim verified against the code as on disk. (1) The cache key is model-free: extract.ts:109-111 `sha256(text + \u0000 + TYPE_SET_VERSION)`. (2) The cache-hit predicate has no model_id: extract.ts:221-224 matches (chunk_id, '__scan__', content_hash, 'ok') only, and the loop (extract.ts:297-326) skips a hit with 0 model calls; deps.modelId (extract.ts:81) is documented as 'recorded on each row; cache is content-keyed' and is written by commitChunk but never read by any lookup. (3) On completion extract_status flips to 'ready' unconditionally (extract.ts:328-332). (4) The sibling tree build IS model-keyed exactly as claimed: tree-build.ts:216-218 `SELECT summary_text FROM summary_cache WHERE content_hash = ? AND model_id = ?`, tree-build.ts:37-39 M12 comment ('a model-switch can't yield a mixed-model tree'), tree-build.ts:75. (5) aggregateExtractions (extract.ts:359-368) GROUPs BY normalized_value with no model_id predicate, so a resume across a model swap yields a silently mixed-model listing — reachable end-to-end: pass interrupted → 'extracting' → reconcileStuckExtracts → 'pending' (ingestion/index.ts:1540-1541) → row action visible again (DocRow.tsx:142-143 requires extractStatus==='ready' to hide) → resumed under runtime.modelId B (handlers/tree.ts:68) with model-A ok scans as cache hits. […]

**Consequences if unfixed.** The flagship 'list every X / how many' coverage-extract answers permanently serve the weakest model that ever scanned the document. A user who upgrades from a 4B to a 27B model and re-runs 'Build deep index' (tree + chained extract) sees the tree re-summarized under the new model (model-keyed cache) but the extraction listing silently unchanged — invisible quality staleness with no refresh affordance.

**Blast radius of a fix.** Contained to the coverage-extract feature's answer quality; no data corruption, no crash. Concrete surface: extractDocument's only production caller is runExtract (apps/desktop/src/main/services/doctasks/handlers/tree.ts:57-83, passing runtime.modelId), enqueued via the #38 tree→extract chain (doctasks/manager.ts:350, :587) and DocumentsScreen.tsx:561-574 (the sole startTask('extract') site, gated hidden by DocRow.tsx:142-143 once both passes are ready). Stale-row consumers: aggregateExtractions (extract.ts:350-421) → listing-answer.ts → the rag:ask coverage-extract route (registerRagIpc.ts:576 extractionsExistInScope) — i.e. every 'list every X / how many' answer permanently reflects whichever model scanned each chunk; a resume across a model swap (interrupt → reconcileStuckExtracts, ingestion/index.ts:1540-1541 → re-run) produces a mixed-model row set the aggregation cannot distinguish. Tests coupled to a fix: whole-doc-extract.test.ts:170-181 (same-model warm cache — must stay 0 calls), :183-220 (DATA-3 marker idempotency), :249-279 (#50 unparsed retry); analysis-extract-hash.test.ts pins contentHashOf byte-identical (frozen digests + do-not-regenerate comment), so any fix MUST be in the marker-lookup predicate, not the hash — exactly as the finder's recommendedFix states. Docs touched by fix-or-accept: rag-design.md §14.5 (line ~1326), extract.ts invariants comment (lines 31-36), known-limitations.md. Cost of the fix: one full re-extract per document on the next explicit run after a model swap (mirrors the SKA-26 version-mismatch posture); existing mixed-model rows on disk persist until then. Note for the fixer: once both statuses are 'ready' the UI offers no re-run, so a marker-predicate fix alone is inert for completed documents until a re-run affordance (or 'stale'-on-model-change marking) also exists.

**Recommended fix.** Add `AND model_id = ?` (the current pass's modelId) to the markerExists cache-hit lookup so a re-run under a different model re-extracts (commitChunk already deletes the chunk's prior rows, so replacement is clean). Do NOT fold the model id into contentHashOf — analysis-extract-hash.test.ts pins that hash byte-identical and existing persisted rows would all orphan; the marker-lookup predicate achieves the same invalidation without breaking the pinned hash.

**Testing needed.** Unit test: scan a chunk under model A (ok marker), re-run extractDocument with modelId B, assert generate() is called again and the rows now carry model B. Regression: same-model re-run stays 0 calls; unparsed-marker retry (#50) unaffected.

**Documentation updates.** architecture.md whole-document-analysis record (extract-pass cache invariants) should state the model-swap invalidation rule; known-limitations.md if the cost (full re-extract per model swap) is accepted instead.

#### F-02 — An in-band SSE error frame mid-stream is silently discarded — a failed generation persists as a clean, complete answer

**Medium · confidence High · reliability · dimension: backend-pipeline · verifier: CONFIRMED**

- **Locations:** `apps/desktop/src/main/services/runtime/llama.ts:110-138` · `apps/desktop/src/main/services/runtime/llama.ts:266-301` · `apps/desktop/src/main/services/chat.ts:1341-1419` · `apps/desktop/src/main/services/translation/completion.ts:104-234`
- **Relation to known/tracked items:** Partial: BUILD_STATE.md:1113-1123 §5 item 7 TS-3(a) registers the real llama-server SSE wire contract — explicitly including "error-frame shapes" — as manual-smoke-only coverage; the defect itself (readChatSSE swallowing in-band mid-stream errors) is NOT registered as known/accepted anywhere (not in docs/known-limitations.md; the §46-§49 ledgers and TA-4 record docs/architecture.md:1825-1858 fixed this class in the translation reader only). Finder's possiblyKnown framing is correct: a concrete consequence of the TS-3(a) gap, not a re-report of it.

**Description.** parseSseLine only recognizes `data:` lines whose JSON carries `choices[0]`; anything else returns {} and is treated as a keep-alive. llama.cpp's server reports a mid-generation failure in-band on the open SSE stream (an `error:`-field event or a `data: {"error":{...}}` frame, depending on build) and then closes the stream without `[DONE]`. readChatSSE treats that close as a normal end (`if (done) break`), so the generator returns cleanly: a partially-generated answer is persisted as if complete — finishReason stays null, so no 'length' badge, no ChatRequestError, no user-visible error. Only a zero-token stream is caught (EmptyCompletionError, CB-4); any partial silently masquerades as a finished reply. The pre-stream non-ok path (chatRequestError) and the process-crash path (fetch rejection + onUnexpectedExit) are handled — only the graceful in-band mid-stream error shape is swallowed.

**Evidence.** llama.ts:117-118 `if (!t.startsWith('data:')) return {}` (an `error:` SSE field line is dropped); llama.ts:121-133 parses only `choices[0].delta/finish_reason` — a `{"error":{...}}` data frame yields `{}`; llama.ts:273 `if (done) break` then the loop exits normally with no error surfaced. chat.ts:1371 `receivedAnyToken` then persists the partial via persistAssistantMessage with `truncated: finishReason === 'length' && capTokens == null` → false.

**Verifier assessment.** Statically verified end-to-end. (1) parseSseLine (runtime/llama.ts:116-137) returns {} for any non-`data:` line (llama.ts:117) — which drops llama.cpp's documented `error: {…}` SSE field shape — and extracts only choices[0].delta/finish_reason (llama.ts:120-133), so a `data: {"error":{…}}` frame also yields {}. (2) readChatSSE treats a stream close as normal end (llama.ts:273 `if (done) break`) with no terminal-frame requirement, returning cleanly. (3) chat.ts:1341-1419: finishReason stays null → truncated=false (chat.ts:1380); non-empty partial content passes the CB-4 empty guard (chat.ts:1394-1397) and persists as a complete answer (chat.ts:1405-1417). Crucially, the wire premise is not speculation — the repo's own translation-audit record (docs/architecture.md:1825-1858, TA-4 M2/M3) states "llama.cpp can emit a mid-stream failure as a bare `error: {…}` SSE field line" and that server-side close mid-decode resolved partials as truncated success, and services/translation/completion.ts:104-234 was hardened for exactly this class (error-field lines → CompletionError, frame.error → throw, close-without-terminal-frame → IncompleteStreamError). The chat reader received none of that hardening; completion.ts's docstring ("structurally mirrors readChatSSE… the terminal-frame + abort contract is stricter here") is descriptive contrast, not a ledgered exemption. […]

**Consequences if unfixed.** A sidecar that fails mid-generation (slot error, KV/context shift refusal, grammar failure, OOM handled gracefully server-side) produces a silently truncated answer presented as complete — the exact 'wrong results in a main flow without a signal' class, in chat, grounded answers, and every doctask generate() (summaries/compare notes cut mid-way with no failure recorded).

**Blast radius of a fix.** All LlamaRuntime.chatStream consumers (verified by grep): chat.ts:1347 (main chat turn — partial persisted complete, truncated=false), chat/compaction.ts:241 (compaction summarizer — a silently truncated summary is written into the conversation's compaction checkpoint, corrupting later turns' context), rag/index.ts:1682 and 1842 (grounded answers), rag/whole-doc-tree.ts:248/474/583 (whole-doc summarize/compare tree — truncated node summaries propagate upward), doctasks/manager.ts:636 (doc-task generate), benchmark.ts:204 (benign), plus vision/runtime.ts:285 which calls readChatSSE directly (image analysis — missed by the finder). Partially self-protected consumers: the JSON-schema flows (skills/categorizer.ts:242, skills/enricher.ts:117, skills/tools/document-edit-locate.ts:150, skills/tools/redaction-locate.ts:153) fail downstream JSON.parse on a truncated payload, converting the silent partial into a visible/handled failure. Mock runtime (mock.ts) never emits error frames — dev/test unaffected; that is exactly why CI stays green (TS-3(a)). Tests touching the area but NOT pinning the swallow: tests/unit/read-chat-sse.test.ts (CB-5 watchdog only), tests/integration/llama-runtime.test.ts (pre-stream HTTP error only, line 251). A fix would change behavior in every flow above (previously-silent partials become surfaced errors) and should mirror completion.ts's TA-4 pattern (error-field + frame.error + missing-[DONE] terminal check), gated on non-abort; chat-stream.ts's existing friendly error mapping is the surface. Docs: architecture.md "Chat & streaming" record and the TS-3(a) inventory row.

**Recommended fix.** In parseSseLine, recognize an error shape: a `data:` JSON with a top-level `error` object (message/type), and optionally non-`data:` `error:` field lines, returning e.g. { streamError }. readChatSSE then throws a ChatRequestError-like error (surfaced by chat-stream.ts's existing friendly mapping) instead of ending cleanly. Verify the exact wire shape against the pinned b9585 server source before implementing.

**Testing needed.** Unit tests on readChatSSE with a fabricated stream: tokens → error frame → close; assert the generator rejects and the partial is not treated as complete. Manual smoke against a real llama-server forced into a mid-stream error (tiny ctx + context-shift disabled) — this is the TS-3(a) manual territory.

**Documentation updates.** architecture.md 'Chat & streaming' record: document the mid-stream error contract once verified against the pinned server.

#### F-03 — prepare-drive.sh --with-assets aborts on stock macOS bash 3.2 (empty-array expansion under set -u, the exact M23 class already fixed in the sibling scripts)

**Medium · confidence High · likely-bug · dimension: build-packaging · verifier: CONFIRMED**

- **Locations:** `scripts/prepare-drive.sh:23` · `scripts/prepare-drive.sh:238-240` · `scripts/prepare-drive.sh:245` · `scripts/prepare-drive.sh:249` · `scripts/fetch-models.sh:161-165` · `scripts/verify-models.sh:90-104`

**Description.** prepare-drive.sh builds COMMON_MODEL_ARGS=() and appends flags only when --accept-license or --dry-run were passed, then expands "${COMMON_MODEL_ARGS[@]}" into each fetch-models call. In the most common invocation (--with-assets with neither optional flag — no license flag is needed because every default-set model is approved) the array is empty, and under `set -u` bash 3.2 (macOS's stock /bin/bash, fixed only in bash 4.4) treats an empty-array expansion as an unbound variable and aborts. The repo explicitly targets bash 3.2 and fixed this exact class in fetch-models.sh/verify-models.sh (M23 guard comment at fetch-models.sh:161: 'Bash 3.2 + set -u: expanding an EMPTY array aborts with unbound variable'), but prepare-drive.sh reintroduces it. Failure scenario: a Mac user runs `scripts/prepare-drive.sh --target /Volumes/X --with-assets`; the tree is laid out, then the script dies with 'COMMON_MODEL_ARGS[@]: unbound variable' before fetching a single model or runtime. The .sh variant's own usage example targets /Volumes/HILBERTRAUM, i.e. macOS is its primary audience.

**Evidence.** scripts/prepare-drive.sh:238-240: `COMMON_MODEL_ARGS=()` / `[[ $ACCEPT_LICENSE -eq 1 ]] && COMMON_MODEL_ARGS+=(--accept-license)` / `[[ $DRY_RUN -eq 1 ]] && COMMON_MODEL_ARGS+=(--dry-run)`; line 249: `bash "$SCRIPT_DIR/fetch-models.sh" --target "$TARGET" --only "$id" "${COMMON_MODEL_ARGS[@]}"` (line 245 same for --all-models); line 23: `set -euo pipefail`. Contrast fetch-models.sh:161-165 where the project documents and guards the identical bash 3.2 semantics for MANIFEST_FILES.

**Verifier assessment.** Verified end-to-end statically. scripts/prepare-drive.sh:23 sets `set -euo pipefail`; lines 238-240 build COMMON_MODEL_ARGS=() appending elements only when --accept-license or --dry-run were passed; lines 245 and 249 expand "${COMMON_MODEL_ARGS[@]}" unguarded. All five DEFAULT_MODEL_IDS manifests carry license_review.status: approved (model-manifests/chat/ministral3-8b-instruct-2512-q4.yaml:39, embeddings/multilingual-e5-small-q8.yaml:31, reranker/bge-reranker-v2-m3.yaml:33, transcriber/whisper-small-multilingual.yaml:34, vision/qwen2.5-vl-3b-instruct-q4.yaml:36), and fetch-models.sh's license gate (fetch-models.sh:15-16) only requires --accept-license for non-approved models — so the natural DIY invocation `--with-assets` alone leaves the array empty. The bash-3.2 semantics claim is asserted by the repo's OWN guard comments for the identical pattern (fetch-models.sh:161 'Bash 3.2 + set -u: expanding an EMPTY array aborts with unbound variable (M23)'; verify-models.sh:99 same; verify-models.sh:90-92 confirms macOS stock Bash 3.2 is an explicit portability target by avoiding mapfile for it), matching the well-documented upstream bug fixed only in bash 4.4. I confirmed locally that bash 5.3 tolerates the empty expansion (exit 0), which explains why the Windows dev box and Linux CI never hit it. […]

**Consequences if unfixed.** On stock macOS bash the advertised 'one command yields a usable drive' DIY flow fails at the fetch step with a cryptic shell error. Linux (bash 4.4+) and the PowerShell sibling are unaffected, which is why CI and the Windows dev box never see it.
