## Executive summary

The OCR backend is in genuinely good shape — the documented invariants (cancel contract,
memory bounds, reply-slot hardening, offline guarantee, encrypted-workspace hygiene) hold in
code and are test-pinned. The feature's problem is at its edges:

1. **The feature is currently unusable from the UI (FE-1, Critical).** A scan-detected
   document is by definition `status === 'failed'`, and the failed-row branch never renders
   the `⋯` overflow menu that holds the only "Make searchable (OCR)" entry point. The scan
   banner tells the user to use a control that never renders. Every downstream piece
   (task handler, progress, completion auto-open) is correctly wired and tested — only
   initiation is dead, and no renderer test covers initiation, which is why it shipped.
2. **The canonical design record misstates the cancel contract (DOC-1, High)** — it still says
   "Cancel persists nothing", the exact claim a prior audit already found wrong and fixed in
   the code header (GAP-7), but the record paragraph was never updated.
3. **Two backend Mediums:** a one-way concurrency guard (BE-1 — doc-task start never checks
   the ingestion `processing` set) and an open packaged-build CSP question (BE-2 — the hidden
   OCR window may effectively run under the dev-relaxed meta-tag CSP in production).
4. A tail of Low findings: stale comments, an undocumented second timeout, a missing page-count
   cap vs the M-2 posture, restart-required-after-asset-install, and UX honesty polish that
   only becomes visible once FE-1 is fixed.

**Interlock warning:** FE-1 (restore initiation), FE-2 (re-run affordance) and BE-1 (start-path
guard) should land together — restoring the UI paths makes BE-1's re-index race user-reachable.

## Critical

### FE-1 — "Make searchable (OCR)" is unreachable; the scan banner points at a control that never renders
*orchestrator-verified (derivation chain, single call site, corroborating test)*

- **Mechanism.**
  1. `scanDetected` is derived as `row.status === 'failed' && row.error_message === PDF_SCAN_DETECTED_MESSAGE`
     (`apps/desktop/src/main/services/ingestion/index.ts:351`) — a scan row is **always** `failed`.
  2. The only OCR entry point is the `showOcr` menu item inside the `⋯` overflow
     (`apps/desktop/src/renderer/screens/documents/DocRow.tsx:356-360`,
     `showOcr = Boolean(d.scanDetected && ocrAvailable)` at `:144`).
  3. The action cluster branches `stateUnknown → rowTask → d.status === 'failed' → else`; the
     `failed` branch renders **only Try again + Remove** (`DocRow.tsx:276-301`), and the
     overflow exists only in the final `else` (non-failed) branch (`:302+`) — where
     `scanDetected` can never be true. The item is dead code.
  4. Meanwhile the scan banner renders `docs.scan.ocrOffer` — *"Use 'Make searchable (OCR)'
     below to read the pages on this drive."* (`DocRow.tsx:204-206`) — instructing the user to
     use the control that never appears.
- **Reachability is airtight:** `onMakeSearchable` has exactly one call site (`DocRow.tsx:357`);
  renderer-wide, `startDocTask` is called only from the translation session and the generic
  `lib/doctasks.ts` helper whose only `'ocr'` caller is that dead item. An existing test even
  pins that failed rows have no overflow
  (`apps/desktop/tests/renderer/DocumentsScreen.test.tsx:296`).
- **Blast radius.** The headline Phase-38 flow cannot be initiated at all — on every OS, dev and
  packaged, plaintext and encrypted. Users see a banner promising a control, find only
  "Try again" (re-parses, fails identically) and "Remove". The completion half
  (auto-open preview after OCR, list refresh — `DocumentsScreen.tsx:539-540`) is correctly
  wired, confirming this is stranded wiring, not intent. `user-guide.md` §"Scanned PDFs —
  'Make searchable (OCR)'" accurately describes the *intended* flow, so the docs currently
  document a flow the app cannot perform (no doc change needed — fix the code).
- **Likely origin (hypothesis, unverified):** the §11.6 refactor that consolidated row actions
  into the `⋯` overflow and deliberately removed the overflow from failed rows stranded the
  OCR item, and no renderer test exercises OCR initiation (see Test gaps).
- **Fix direction.** Render a "Make searchable (OCR)" affordance in the failed branch when
  `showOcr` (a primary inline button beside Try again is most honest), or model scan-detected
  as a non-failed sub-state. Add the missing renderer test (scan row → click → `startTask('ocr')`).

## High

### DOC-1 — architecture.md's OCR design record still claims "Cancel persists nothing", contradicting the deliberate GAP-7 contract
*orchestrator-verified*

- **Doc:** `docs/architecture.md:1567` (the Phase-38 design record): "…re-running the task is
  the explicit redo. **Cancel persists nothing.**"
- **Code truth:** `apps/desktop/src/main/services/doctasks/handlers/ocr.ts:24-31` — the GAP-7
  contract (full-audit 2026-07-11, decided deliberately): a cancel **before** the persist point
  persists nothing; a cancel **after** it (during the signal-less, minutes-long re-ingest) is
  deliberately ignored and the task completes `'done'`. Test-pinned:
  `tests/integration/ocr-task.test.ts` ("GAP-7: a cancel during the re-ingest completes as
  'done'…").
- **Aggravation:** the audit ledger later in the same file (~`architecture.md:7066`) records
  that this exact blanket claim was found stale and the *code header* rewritten — but the
  canonical record paragraph 5,500 lines earlier was never updated. A contributor reading the
  record (the normal path) learns the admitted-wrong contract.
- **Fix direction.** One-line edit: replace "Cancel persists nothing." with the two-part GAP-7
  contract in the record paragraph.

## Medium

### BE-1 — Doc-task start bypasses the ingestion `processing` guard: an OCR re-run can race a concurrent re-index of the same document
*orchestrator-verified (one-way guard confirmed)*

- **Mechanism.** `registerDocsIpc` refuses re-index/delete while a task targets the doc
  (`requireNoActiveTask`, `apps/desktop/src/main/ipc/registerDocsIpc.ts:224`), but the reverse
  direction doesn't exist: `startDocTask({kind:'ocr'})` validates only
  `!doc || !isPdfPath(doc.title) || !(doc.scanDetected || doc.ocr)`
  (`apps/desktop/src/main/services/doctasks/manager.ts:306-312`) and never consults the
  module-local `processing` set (`registerDocsIpc.ts:735`). OCR is the uniquely exposed kind:
  every other kind requires `status === 'indexed'`, which a running re-index flips away, but
  an **already-OCR'd** PDF keeps `ocr` metadata for the whole re-index (by design), so an OCR
  task admitted mid-re-index interleaves two chunk DELETE/INSERT transactions + two embed
  phases on one document. The team knows doc-task ingestions run outside `processing` — the
  reconcile sweep is patched for it (`registerDocsIpc.ts:806`, DB-3) — but the start-side hole
  remains.
- **Blast radius.** Duplicated/orphaned chunks and embeddings, retrieval degraded until a clean
  re-index; recoverable, no crash. **Today it needs a devtools-level IPC call** (the UI can't
  start OCR at all per FE-1, and couldn't start it on indexed docs even before) — it becomes
  user-reachable exactly when FE-1/FE-2 restore initiation and re-run. Fix together.
- **Fix direction.** Thread an `isDocumentProcessing(id)` dep into `DocTaskDeps` and refuse
  `startDocTask` for a doc in `processing`/an active import job — the mirror of
  `requireNoActiveTask`. Add the missing concurrency test.

### BE-2 — Packaged-build CSP for the hidden OCR window may effectively be the meta tag, which allows `connect-src localhost:*` in production
*needs-verification (mechanism confirmed; packaged behavior unmeasured)*

- **Mechanism.** The strict prod CSP (`connect-src 'self'`) is injected only as a response
  header via `session.webRequest.onHeadersReceived` (`apps/desktop/src/main/index.ts:474-475`).
  Both windows load over `file://` in packaged builds, and per Electron's documented guidance
  header CSP delivery is unavailable for `file://` — the reason the meta tags exist. The
  shipped fallback meta in `apps/desktop/src/renderer/ocr.html:12` bakes in the dev
  relaxation: `connect-src 'self' ws://localhost:* http://localhost:*` (same in `index.html`).
  The hidden OCR window is the one context rendering hostile bytes (untrusted PDFs through
  pdfjs); under the meta-only policy a pdfjs compromise could reach the llama.cpp/embedder/
  translator sidecars on 127.0.0.1 despite sandbox + navigation guard. Remote origins stay
  blocked either way — this is defense-in-depth erosion, not exfiltration.
  `docs/security-model.md` (~:36-41) tells the strict-header story, which is then not the
  load-bearing mechanism on `file://`.
- **Follow-up to settle it (single manual step):** launch a packaged build and inspect the
  effective CSP in both windows. If the header *does* attach on `file://`, this downgrades to
  a doc nit — but then check the intersection doesn't strip `blob:` workers pdfjs needs.
- **Fix direction.** Emit a strict prod meta at build time (or template the localhost entries
  dev-only), and reconcile `window-security.ts` + security-model.md with whichever mechanism
  is actually load-bearing.

### FE-2 — No re-OCR affordance, though the backend explicitly supports re-runs
*orchestrator-verified (backend intent + UI gate)*

- The manager admits "an already-OCR'd PDF being re-run (better assets / a bad first pass)"
  (`manager.ts:307-308`), and the ingestion comment says the OCR task targets scan rows "plus
  already-OCR'd PDFs for a re-run" (`ingestion/index.ts:349-350`); the design record calls
  re-running "the explicit redo" (D33). The UI gate `showOcr = scanDetected && ocrAvailable`
  (`DocRow.tsx:144`) never offers it for an OCR'd doc — a user with a bad first recognition
  has no path but Re-index, which by design *reuses* the stored pages. The "Scanned / OCR"
  smart view even lists OCR'd docs (`shared/types.ts:1556`) with no redo present.
- **Fix direction.** Gate the item on `(d.scanDetected || d.ocr != null) && ocrAvailable`
  (label the redo case distinctly, e.g. "Read again (OCR)"). Coordinate with BE-1.

### FE-3 — Translate mislabels a scanned PDF "unsupported file type"; no scan message or OCR handoff
*orchestrator-verified*

- A fully image-only PDF imports fine in Translate, then ingestion fails (scan detected);
  `runImport` collapses **every** imported-but-ingest-failed case to `fail('unsupported')`
  (`apps/desktop/src/renderer/lib/fileTranslateSession.ts:305-308`), rendering
  *"That file type can't be translated. Try a PDF, Word, Markdown, or text file."*
  (`shared/i18n/en.ts` `translate.file.err.unsupported`) — nonsensical for a file that *is* a
  PDF, with no pointer to "Make searchable (OCR)" in Documents. Unlike FE-1 this path is fully
  reachable today; encrypted/corrupt PDFs get the same misleading copy.
- **Fix direction.** Surface the document's localized `error_message` for this case and
  special-case scan detection with a "make it searchable in Documents first" hint.

### DOC-2 — User-facing docs are silent on the post-persist cancel behavior
*agent-verified (code side orchestrator-verified via DOC-1)*

- `known-limitations.md` (~:1875-1879) documents the mid-page cancel/timeout case in detail;
  `user-guide.md` (~:362, :397-399) says "You can always cancel" — but nothing user-facing
  says a cancel landing during the final re-index step is deliberately ignored and the
  document completes searchable. A user who cancels late and finds the doc searchable anyway
  has no documented explanation. (This is the honest-UX counterpart of DOC-1; see also FE-5.)
- **Fix direction.** One bullet in known-limitations.md's OCR section beside the mid-page one;
  soften user-guide's "always".

### DOC-4 — The documented "2-minute, env-tunable per-page ceiling" is only half the story: a fixed 60 s render-step timeout is undocumented
*orchestrator-verified (code side; doc quotes agent-verified)*

- Docs (`known-limitations.md` ~:1876-1878, `architecture.md` ~:1547-1556) describe one
  per-page ceiling: 2 minutes, `HILBERTRAUM_OCR_PAGE_TIMEOUT_MS`. The rasterizer has a second,
  hard-coded one: `RASTER_STEP_TIMEOUT_MS = 60_000` bounding the open and per-page **render**
  round-trips (`apps/desktop/src/main/services/ocr/rasterizer.ts:51`), no env override. A page
  that renders slowly fails the task at 60 s — under the documented figure, and tuning the
  documented env var does nothing for it.
- **Fix direction.** Document the second ceiling (and consider an env override for symmetry).

### DOC-5 + BE-4 (merged) — OCR assets installed mid-session require a restart: undocumented, and the code comment claims the opposite
*agent-verified by two agents independently; mechanism consistent across both reports*

- The engine is composed once at startup (`compose-services.ts:159-162`;
  `main/index.ts:291` closes over the const; `AppStatus.ocrAvailable` at
  `registerCoreIpc.ts:59` reads that const; the renderer reads it once at mount,
  `DocumentsScreen.tsx:291-294`). Nothing re-runs `createSelectedOcrEngine` — unlike the
  translator's `onModelInstalled` re-composition (issue #40 precedent).
- Two deviations from that reality: (a) `doctasks/context.ts:54-59` says "Read per task (the
  assets can appear mid-session)" — the getter is read per task but always returns the boot
  const, so the claim misleads; (b) `troubleshooting.md` (~:190-193) tells the user how to
  fetch the missing OCR files but never says to restart the app afterward — the offer will
  not appear until relaunch, and neither user-guide, drive-layout, nor packaging says so.
- **Fix direction.** Either mirror the translator's refresh for the `ocr` role, or fix the
  comment and add the restart sentence to troubleshooting.md/drive-layout.md.

## Low

### BE-3 — Stale "Backpressure" comment contradicts the shipped ING-5 look-ahead
`handlers/ocr.ts:52-53` says "the next page is not rendered until this recognition ends";
since ING-5, `pipelinePages` renders N+1 *while* N recognizes (`pipeline.ts:27-35`) — the real
invariant is "at most one extra PNG resident" (correctly stated in `rasterizer.ts:33-38`).
Comment-only; misleads memory/cancellation reasoning. *(agent-verified, certain)*

### BE-5 — Tesseract `stop()` claims and guards are half wrong
`tesseract.ts:19` says stop happens on "will-quit / lock", but the workspace-lock teardown
list (`registerWorkspaceIpc.ts:280-298`) omits the OCR engine — an idle warm WASM worker
survives a lock (RAM-only; breaks stated sidecar parity). And `ensureWorker()`
(`tesseract.ts:111`) never checks `this.stopped`, so a queued recognition could lazily respawn
a worker after teardown — latent today (stop is quit-only). Fix: `stopped` guard in
`ensureWorker`; either add the engine to lock teardown or correct the comment.
*(agent-verified, certain; latency of (b) noted)*

### BE-6 — No page-count cap: OCR rasterizes every declared page, unlike ingestion's M-2 posture
The parser walks at most `pdfMaxPages` (5000) because "a tiny PDF can declare an enormous page
count"; the OCR pipeline loops `1..pageCount` straight from pdfjs with no clamp
(`rasterizer.ts:216-218`). Memory stays bounded and the task is cancellable with visible
progress, so this is an attended CPU sink — but the cap asymmetry has no recorded decision.
Fix: clamp to `pdfMaxPages` (log truncation) or record the exemption. *(agent-verified, likely)*

### BE-7 — Hidden-window pdfjs caches never released across a long scan
`renderer/ocr/main.ts:41-59` never calls `page.cleanup()`; an image-heavy multi-hundred-page
scan can grow the hidden renderer until an OOM turns each remaining step into a 60 s timeout
stall (bounded, non-leaking — window destroyed per task; no `render-process-gone` fast-fail).
Fix: `page.cleanup()` after each page (cheap); optionally subscribe `render-process-gone`.
*(needs-verification: whether pdfjs v6's internal eviction keeps this flat — manual-smoke territory)*

### BE-8 — `ImageParser` ignores the parse context's abort signal
`parsers/image.ts:49` calls `engine.recognize(image)` without `{ signal: ctx?.signal }` — a
cancelled import waits out the recognition (bounded by the 2-min ceiling; photos are seconds).
One-line fix. *(agent-verified, certain)*

### BE-9 — No CVE-review cadence for pinned parser libraries; engine id hardcodes the version
pdfjs parses untrusted bytes in the main process *and* the hidden window; it's pinned only via
the lockfile, and no doc flags pdfjs/tesseract as needing periodic advisory review — which
matters because the offline posture removes every auto-update channel (pdf.js has prior
renderer-code-execution history, e.g. the CVE-2024-4367 class). Separately
`tesseract.ts:108` hardcodes `id = 'tesseract.js-7.0.0'` — a future bump silently stamps wrong
provenance into `ocr_json.engineId`. Fix: a security-model.md line + derive the id from the
package version. *(agent-verified)*

### FE-4 — Progress label says "Reading the scan…" through the re-ingest step
`stepsTotal = pages + 1` (`handlers/ocr.ts:49-51`); the row renders
`docs.task.ocrBusy (stepsDone/stepsTotal)` (`DocRow.tsx:146-152`) — so a 3-page scan shows
"Reading the scan… (4/4)" during the minutes-long re-index. The Translate surface subtracts
the final step for display (`fileTranslateSession.ts:159-165`); the two are inconsistent.
Latent until FE-1 is fixed. *(agent-verified, certain)*

### FE-5 — Cancel stays enabled through the deliberately-uncancellable window with no feedback
During the post-persist re-ingest the backend no-ops cancels (GAP-7), but the row's Cancel
button (`DocRow.tsx:268-274`) stays enabled and clicking it silently does nothing — the task
then completes and auto-opens the preview. Defensible outcome, but an enabled "Stop" that
won't stop is a feedback gap (pairs with DOC-2). Latent until FE-1. Fix: disable/relabel
("Finishing…") once past persist. *(agent-verified; latent)*

### DOC-3 — "Recognized text survives re-index" is PDF-only; photos re-OCR every re-index
The known-limitations bullet (~:1869-1872) sits under the heading covering both PDFs and
photos, but the `ocrPages` reuse hook is gated `isPdfPath(row.title)`
(`ingestion/index.ts:772-773`) — a photo re-recognizes from scratch on every re-index
(`parsers/image.ts:13-15`, by design, seconds). Add the disclosing clause. *(agent-verified)*

## Verified clean (the load-bearing properties that hold)

Backend (all orchestrator-spot-checked or test-pinned per the backend agent's citations):
- **Hidden-window hardening:** dedicated tiny preload + `SECURE_WINDOW_WEB_PREFERENCES`
  (contextIsolation/sandbox/no-node), window-open denied, `installNavigationGuard(…, () => false)`
  covering `will-navigate` + `will-redirect` (`rasterizer.ts:121-140`); string-pinned by
  `window-security.test.ts`.
- **Reply spoofing blocked:** replies accepted only from `win.webContents`
  (`rasterizer.ts:149`); the five `OCR_RASTER` channels are absent from the app preload; the
  page number fed to `onPage` is main's own loop counter.
- **Lifecycle:** one `finally` resets the in-use latch, removes the abort listener and all
  `ipcMain` listeners, destroys the window on every exit path (`rasterizer.ts:233-238`).
- **GAP-7 cancel contract genuinely upheld** with no await between the last pre-persist check
  and `setDocumentOcr` (`handlers/ocr.ts:69-85`); both halves integration-tested.
- **Memory bounds:** ING-5 1-deep look-ahead + R4 single-drain (`pipeline.ts:25-48`) and the
  REL-4 96 MiB page byte cap enforced before recognition (`page-cap.ts`), unit-tested.
- **Offline guarantee:** every tesseract option explicit and local (`langPath` on the drive,
  `cacheMethod:'none'`, explicit `workerPath`), a no-CDN sentinel test greps all of `src/`;
  pdfjs worker is the bundled asset; downloads are dev/packaging-time with sha256 verification.
- **Packaged resolution:** `asarUnpack` for tesseract.js/-core + `app.asar → app.asar.unpacked`
  rewrite for both path separators; no hardcoded developer paths in the OCR chain.
- **Persistence integrity:** `ocr_json`/`ocr_meta_json` written and cleared in lock-step;
  tolerant parses; list projection excludes the big blob; crash between persist and re-ingest
  is recoverable (stored recognition re-used via the `ocrPages` hook on the next re-index).
- **Encrypted workspaces:** decrypt-to-`<id>.parse-ocr.pdf`, shredded in `finally`, matches the
  startup crash-sweep pattern; OCR text goes to the DB column only; audit events carry ids only.
- **Consumers need no special-casing:** retrieval/citations flow through ordinary chunks;
  preview/translation hit the same `ocrPages` hook (with the issue-#58 page accounting);
  evidence-pack source context reads stored chunks only ("NEVER a re-read/re-parse/re-OCR").

Frontend:
- **EN/DE parity holds for every OCR key** on the audited surfaces (test-enforced), German
  strings genuinely German; `ocrInfo` pluralization correct; the scan banner localizes the
  persisted-English notice through the display map; `main.task.*` errors are throw-site
  localized via `tMain` (their absence from the display map is correct, not a bug).
- **Completion wiring is sound:** a finished OCR task refreshes the list and auto-opens the
  searchable preview (`DocumentsScreen.tsx:539-540`).
- **ImagesScreen is correctly OCR-free** (vision Q&A; its unavailable-card pointer to
  Documents' OCR is accurate); Home has no OCR entry point; Translate's #58 gap warnings
  (missing pages / failed parts) render honestly in both languages.

Docs (docs agent's verified-accurate list, spot-checked): scan-notice string byte-identical
end to end; hybrid-PDF non-detection accurately documented; 300 DPI/4096 px render target,
five `OCR_RASTER` channels, 2-min recognition timeout + env var, availability-driven engine
(D14/D9), no `ocrLanguages` settings key, sha256s in model-policy.md matching
`runtime-sources.yaml` character-for-character, asarUnpack documented correctly,
notices/licenses (THIRD-PARTY-NOTICES tesseract.js@7.0.0, DRIVE-NOTICES `ocr 4.0.0_best_int`,
licenses/README mapping) all consistent.

## Method & scope notes

- Sub-agent reports were merged with de-duplication (BE-4 and DOC-5 converged independently on
  the same root cause — merged). Every Critical/High/Medium mechanism was re-verified by the
  orchestrator against the working tree except BE-2's packaged-build behavior (named follow-up)
  and the doc-side line numbers of DOC-2/DOC-4 quotes (code side verified).
- Not exercised: the real hidden-window rasterizer (manual smoke; needs real assets), packaged
  builds (BE-2, BE-7 confirmation), recognition quality (R-O3 figures taken as recorded).
